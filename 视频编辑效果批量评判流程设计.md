# 视频编辑效果批量评判流程设计文档

## 1. 任务定义

**输入**：一批视频文件，每个视频画面内嵌三部分内容：
- 顶部文字：编辑指令（Prompt），如"调整灯光，模拟金色时刻，营造暖色调和柔和阴影"
- 左侧画面：edit_1
- 中间画面：src（原图）
- 右侧画面：edit_2

**输出**：每个视频归类到三个类别之一：
- `edit_1`：edit_1 更好地执行了指令
- `edit_2`：edit_2 更好地执行了指令
- `平局`：两者执行效果相当，或都不理想/都很好，难分伯仲

**下游**：分类结果对接你现有的 AHK 脚本目录结构：
```
E:\Work_lyl\concat_disagree\edit1
E:\Work_lyl\concat_disagree\edit2
E:\Work_lyl\concat_disagree\平局
```

## 2. 模型选型建议

你截图里的候选模型：`qwen3.6-35b`、`qwen3-vl-8b`、`qwen3-vl-30b`、`gemma-4-2...`

| 模型 | 适用场景 | 建议 |
|---|---|---|
| **qwen3-vl-30b** | 精细视觉对比、中文指令理解 | ✅ **主力模型**。参数量大，对细微光影/色调差异的判断稳定性明显优于8B版本，中文OCR和指令理解也更准 |
| qwen3-vl-8b | 快速初筛、大批量粗筛 | 可用作第一轮快速筛选（比如先筛出"明显平局"的简单样本），减轻30B的负载 |
| qwen3.6-35b | 纯文本任务 | 注意确认它是否支持视觉输入（多模态）。如果不支持图像，此任务用不上，可以留作后续做"结果复核报告生成"等纯文本环节 |
| google/gemma-4-26b-a4b-qat | 备用/交叉验证 | 若为多模态版本，可作为**第二意见模型**，用于交叉验证 qwen3-vl 的判断（两模型意见一致的样本直接采信，不一致的转人工） |

**核心建议**：以 `qwen3-vl-30b` 作为主评判模型，重要样本可选配一个不同架构的模型做交叉验证，提高判断置信度。

### 2.1 LM Studio 调用方式说明

LM Studio 本地服务默认走 **OpenAI 兼容接口**（不是 Ollama 的 `/api/generate` 格式），调用时需要注意：

- **启动方式**：LM Studio 里加载好模型后，开启"Local Server"（默认端口 `1234`），会得到形如 `http://localhost:1234/v1/chat/completions` 的接口
- **图片输入格式**：走 `chat/completions` 接口，图片以 base64 塞进 `content` 数组，格式类似：

```json
{
  "model": "qwen3-vl-30b",
  "messages": [
    {
      "role": "user",
      "content": [
        {"type": "text", "text": "你的Prompt文本"},
        {"type": "image_url", "image_url": {"url": "data:image/jpeg;base64,{base64编码}"}}
      ]
    }
  ],
  "temperature": 0.2
}
```

- **model字段**：填 LM Studio 里显示的模型标识（一般就是列表里那个名字，如 `qwen/qwen3-vl-30b`，具体以 LM Studio "Local Server" 页面给出的示例代码里的名称为准，它有时会跟侧边栏显示的名字略有出入）
- **temperature 建议调低**（如 0.1-0.3）：这是判断/评审类任务，不需要模型有创造性发挥，低温度能让同一样本的判断更稳定，减少无谓的随机波动干扰"一致性校验"的判断
- **批量并发注意**：LM Studio 本地服务默认是**单请求串行处理**，如果你的脚本开多线程并发调用，大概率不会提速（甚至可能报错或排队），批量跑的时候按顺序串行调用即可，效率瓶颈主要在模型推理本身而不是并发数

## 3. 整体流程（Pipeline）：模型初筛 + 人工复核

核心思路：**模型负责把绝大多数"明显好判断"的样本一次性筛完并自动归档，只把少数"模型自己也拿不准"的样本挑出来交给人工**。这样人工只需要看一小部分样本，而不是从头到尾看完所有视频。

```
┌─────────────┐   ┌──────────┐   ┌────────────┐   ┌──────────────┐
│ 视频批量输入 │ → │  抽帧    │ → │  模型初筛   │ → │ 按置信度分流  │
└─────────────┘   └──────────┘   └────────────┘   └──────────────┘
                                                          │
                                    ┌─────────────────────┴─────────────────────┐
                                    ▼                                           ▼
                          高置信度 → 自动归档                          低置信度/有分歧 → 人工复核队列
                       （edit1/edit2/平局目录）                          （人工看完后手动分类，
                                                                          复用现有AHK脚本Ctrl+1/2/3）
```

**"置信度"如何判定**：不是让模型自己喊"我有90%把握"（这类自评分不太可靠），而是靠**模型对同一样本判断两次、看是否一致**来间接衡量置信度（具体方法见第5节的位置交换）。两次判断一致 → 视为高置信度，直接自动归档；两次判断不一致 → 视为低置信度，进入人工复核队列。

这样设计的好处：
- 人工只需要处理"模型自己都纠结"的疑难样本，工作量大幅减少
- 复用你现有的 AHK 脚本（Ctrl+1/2/3 快捷键）来做人工复核阶段的最终分类，不用额外学新工具
- 初筛阶段全自动跑批，可以挂后台跑，不占用人工时间

### 步骤 1：抽帧
- 由于画面内容是静态合成对比图（src/edit_1/edit_2 + 文字），**不需要多帧**，抽 1 帧即可代表全部信息
- 建议抽取视频**中间时间点**的帧（避免开头/结尾可能存在的转场黑场或淡入淡出）
- 如果怀疑存在动态效果展示（比如视频过程中效果在变化），再考虑抽 3 帧（首/中/尾），取多数结果

### 步骤 2：VLM 判断
- 单帧输入模型，使用第 4 节设计的 Prompt
- 模型输出：**分析过程 + 结构化结论**

### 步骤 3：模型初筛（含一致性自检，解决 VLM 常见的"位置偏差"问题）
VLM 在做 A/B 比较时，容易偏向"排在前面/左边"的选项，同一张图片如果调换 edit_1 和 edit_2 的位置重新问一遍，结果可能不一致。用这个特性来做初筛的置信度判断：

- 对每个样本跑 **2 次判断**：
  1. 原始帧（edit_1 在左，edit_2 在右）
  2. 左右镜像翻转后的帧（或用图像处理软件交换 edit_1/edit_2 位置），重新贴上正确标签再问一次
- 若两次结果一致 → 判为**高置信度**，初筛阶段直接给出结论
- 若两次结果不一致 → 判为**低置信度**，初筛阶段不下结论，直接转人工

### 步骤 4：按置信度分流
- **高置信度样本**：脚本自动移动/复制到对应目录（edit1 / edit2 / 平局），全程无需人工介入
- **低置信度样本**：连同模型两次判断的完整分析过程一起，放入 `_待人工复核` 目录，人工打开视频后用现有 AHK 脚本（Ctrl+1/2/3）做最终裁决。模型的分析文字可以作为人工判断的参考依据，加快人工复核速度（不用从零看起，可以直接看模型纠结在哪一点）

## 4. Prompt 设计

### 4.1 设计原则
- **强制思维链**：先分析再下结论，避免模型"跳过思考直接猜"
- **结构化输出**：最后一行给固定格式的结论，方便脚本解析
- **明确评判维度**：不要让模型泛泛而谈，要求它对照指令的具体要求逐项检查

### 4.2 Prompt 模板

```
你是专业的图像编辑质量评审员。图片中包含三部分：
- 左侧标注"edit_1"的图像
- 中间标注"src"的原始图像
- 右侧标注"edit_2"的图像
- 顶部文字是编辑指令

编辑指令：{从图中OCR提取，或程序传入的原始指令文本}

请按以下步骤评审：

1. 【指令拆解】将指令拆解为具体的可验证要求（例如："金色时刻光影"可拆解为：暖色调、低角度光源、柔和阴影、天空颜色变化等）
2. 【edit_1分析】逐项对照上述要求，说明edit_1相对src的变化，以及是否达成、达成程度如何
3. 【edit_2分析】同上，分析edit_2
4. 【对比结论】直接比较两者：谁更贴合指令？是否存在其中一个过度编辑/编辑不足/引入了不该有的瑕疵（如失真、不自然的光斑、色偏过度）？

最后，另起一行，严格按以下格式输出结论（不要有多余文字）：
结论：edit_1
或
结论：edit_2
或
结论：平局
```

### 4.3 结果解析
脚本从输出中用正则提取最后一行 `结论：(edit_1|edit_2|平局)`，若解析失败（模型没有按格式输出），该样本自动归入人工复核队列。

## 5. 两次判断的位置交换方法

具体到你的画面（左中右三联图）实现"交换位置"有两种方式：

**方式A：图像处理交换**
用 PIL/OpenCV 把画面裁切为左右两块（edit_1区域、edit_2区域），交换位置后重新拼接，文字标签也要跟着换（"edit_1"标签移到右边，"edit_2"标签移到左边），生成第二张测试图。

**方式B（更省事）：仅在Prompt里交换称呼，不改图像**
不改图像本身，而是在prompt里告诉模型："现在请把图中左侧的图像称为B，右侧的图像称为A"，用这种方式测试模型是否会因为"称呼顺序"或"文字标签"而产生偏向。

推荐用**方式A**，因为它测试的是真正的位置偏差（模型对"左边/右边"这个空间位置本身的偏向），比单纯换称呼更能反映真实的偏差情况。如果图像裁剪拼接实现起来麻烦，方式B可以作为简化版先跑起来。

## 6. 质量控制与验证

在正式批量跑之前，建议：

1. **小样本人工标注基线**：先人工标注 50-100 个视频的"正确答案"，作为验证集
2. **跑通pipeline验证准确率**：用同一批样本跑模型判断，对比人工标注，计算准确率/一致率
3. **根据结果调整**：
   - 如果准确率不理想，优先检查 Prompt 是否需要补充更具体的评判标准（比如针对"金色时刻""柔和阴影"等常见指令类型，给出更明确的判断细则）
   - 如果 8B 模型准确率明显低于 30B，全量跑时优先用 30B，8B 只做初筛
4. **定期抽检**：批量跑的过程中，定期抽取一定比例（如5%）的"高置信度"结果做人工抽检，监控模型是否随不同指令类型出现系统性偏差
5. **监控人工复核比例**：正常情况下低置信度（转人工）样本应该是少数。如果某一批次转人工比例明显偏高（比如超过30%），说明这批指令类型可能超出模型能力范围，需要针对性优化Prompt，而不是让人工一直兜底

## 7. 批处理与目录设计（衔接现有AHK脚本）

建议输出目录结构：

```
E:\Work_lyl\concat_disagree\
├── edit1\              ← 高置信度判定为edit_1更好
├── edit2\              ← 高置信度判定为edit_2更好
├── 平局\                ← 高置信度判定为平局
├── _待人工复核\          ← 两次判断结果不一致，或解析失败
└── _判断日志\
    └── results.csv     ← 记录：视频文件名、指令文本、两次判断详细过程、最终结论、是否一致
```

`results.csv` 建议字段：

| 字段 | 说明 |
|---|---|
| video_name | 视频文件名 |
| instruction | 提取的指令文本 |
| judge1_raw | 第一次判断完整输出 |
| judge1_result | 第一次判断结论 |
| judge2_raw | 第二次（交换位置后）判断完整输出 |
| judge2_result | 第二次判断结论 |
| consistent | 两次是否一致 |
| final_label | 最终归类结果 |
| model_used | 使用的模型名称 |

保留完整日志的好处：后续如果发现模型判断有系统性问题，可以直接从 CSV 里筛选、复盘，不需要重新跑视频。

## 8. 后续可扩展方向

- **多模型交叉验证**：qwen3-vl-30b 与 gemma（若支持多模态）分别判断，只有两者一致时才算高置信度，进一步减少误判进入自动归档
- **按指令类型分桶统计准确率**：例如"光影类指令"和"色彩类指令"分开统计模型表现，找出模型的薄弱指令类型，针对性优化prompt
- **难例样本库**：把人工复核中反复出现分歧的案例整理成"难例集"，用于后续升级模型或换prompt策略时做回归测试
